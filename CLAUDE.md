# BawrSpam — Personal WoW Chat-Spam Filter

Retail-only WoW addon. Filters chat (Trade / General / Yell / Say / Whisper /
Emote), LFG listings + applicants, and chat bubbles. Every block is captured
in a History panel with one-click false-positive recovery.

Current status: **pre-implementation.** Design spec FINAL; no code yet.

## Critical rules

1. **Original work only.** Inspired by funkydude's BadBoy (which pioneered
   Unicode-confusable folding + weighted scoring for WoW chat filtering),
   but NO BadBoy code, data, or pattern lists are copied. Independent
   clean-room implementation. See spec "Inspiration & clean-room boundary"
   section for the tier table and attribution language.

2. **Pattern data never leaves the private repo.** `spam_master.txt`,
   `fixtures.lua`, and `.salt` stay in `BawrSpam_Dev/`. Only the encoded
   `PatternData.lua` ships in the public repo. Loss of the public repo
   does not expose pattern source.

3. **`.salt` is never committed.** Even inside `BawrSpam_Dev/`, the salt is
   gitignored. Loss of the salt breaks ruleId stability across builds; back
   it up out-of-band.

4. **Cleanse has zero WoW API references.** It must run identically in the
   addon (loaded via TOC) and in the build tool (loaded via `dofile`). The
   dual-mode pattern is spec §4.2.

5. **ChatScanner filter install is single-shot per event.** The
   `filterInstalled[event]` guard (spec §6.1, added in Codex round 9) is
   load-bearing — without it, the PLAYER_LOGIN call and the +1s reassert
   stack duplicate filter functors on the chat-event chain.

6. **Trust uses `IsPlayerInGuildFromGUID(guid)`, NOT `IsGuildMember(name)`.**
   `IsGuildMember` takes a NAME string and is the wrong call for a GUID-keyed
   trust check (MCP-confirmed, round 8). Three places in the spec corrected
   in round 9 — do not regress.

7. **LFG render-hide has a ScrollBox row-recycling contract.** Hidden rows
   stay hidden when recycled for a different ID unless explicitly re-shown.
   Every sweep / hook pass must re-assert visibility from the current
   `IsLFG*Blocked(row.id)` result, never from prior state. See spec §9.7.

8. **All non-trivial work uses a git worktree branch.** Convention:
   `.worktrees/BSP-{id}-{name}` inside the repo (matches Triage, not
   Homestead's sibling pattern).

9. **Co-author line on every commit:**
   `Co-Authored-By: Royaleint and Claude Code`.

## MCP usage

Platform WoW addon knowledge via the `wow-api` MCP server. Most-relevant
tools for BawrSpam:

- `lua51_rules`, `api_conventions`, `ace3_patterns`, `performance_rules`
- `lookup_api`, `lookup_blizzard_symbol`, `find_blizzard_api_call`,
  `find_blizzard_event_usage`
- `search_blizzard_ui` — load-bearing for §9.7 LFG render-hide hook
  discovery and the chat-name dropdown right-click investigation (§5.7)
- `check_taint` — for any new API touch outside the confirmed-safe set
- `mcp_status`, `reference_health` — when validating freshness

BawrSpam is not yet registered with the wow-api MCP `project_*` tools
(no `project_reference("bawrspam", …)` overlay yet). Add the overlay
in v1.1+ once the codebase is stable.

## Tier routing (inherits from `../BawrLabs/CLAUDE.md`)

- **Tier 0** — ≤1 file, ≤25 lines, no SavedVariables / no WoW API uncertainty.
- **Tier 1** — ≤3 files, ≤100 lines, localized behavior.
- **Tier 2** — ownership, scanner, vendor data, taint, release, SavedVariables,
  multi-module behavior, or third-party integrations. Full PLAN_TEMPLATE +
  Argus + Gate 2.

The chat-event hot path, Trust pipeline, SavedVariables schema, build tool,
and LFG render-hide are all Tier 2. Most UI tweaks and pattern-corpus
additions will be Tier 1.

## The Two Gates

- **Gate 1 — Argus**: code quality, correctness, performance, philosophy.
  Runs on uncommitted worktree before merge.
- **Gate 2 — Rawb**: in-game functional verification after merge. Only Rawb
  clears this.

## Design philosophy

- **Transparency by default.** Every block has an audit trail; no silent
  filtering. The History panel is the recovery contract — restore is
  always one click away.
- **Original message preserved at block time.** History is the recovered
  view; we do NOT re-inject blocked messages into the chat frame (avoids
  ordering / coloring / timestamp ambiguity).
- **Detection-only is not shipping.** Per spec §9.7, LFG must visually
  hide rows; if hook discovery fails, LFG defers to v1.1 entirely. We do
  not ship a feature that just echoes what the user already saw.
- **Patch-churn is silent.** `geterrorhandler()` calls gate on
  `Trust._devMode`. Loud errors during normal use are reserved for genuine
  init failures, not menu-hook or render-hook patch breakage.
